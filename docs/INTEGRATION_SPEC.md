## 1. PDA Derivation

In ORIN, a user's identity is entirely abstracted via an email hash. The Anchor contract stores the `GuestIdentity` as a PDA (Program Derived Address). You don't need the user to remember a specific wallet address; you just need their hashed email.

**Seeds:** `[b"guest", sha256(email)]`

###  TypeScript Implementation:
```typescript
import { PublicKey } from '@solana/web3.js';
// Make sure to install js-sha256 via npm/yarn
import { sha256 } from 'js-sha256';

// The Program ID generated by our Anchor backend
const ORIN_PROGRAM_ID = new PublicKey("FqtrHgdYTph1DSP9jDYD7xrKPrjSjCTtnw6fyKMmboYk");

/**
 * Derives the guest's on-chain Profile PDA based on their email.
 * @param email - The user's raw email string
 * @returns The PDA PublicKey
 */
export function getGuestAddress(email: string): PublicKey {
  // We use exactly 32 bytes for the hash as defined in our Rust contract
  const emailHashArray = sha256.array(email.toLowerCase().trim());
  const emailHash = Buffer.from(emailHashArray);
  
  const [guestPda] = PublicKey.findProgramAddressSync(
    [Buffer.from("guest"), emailHash],
    ORIN_PROGRAM_ID
  );

  return guestPda;
}
```

## 2. Core Data Structures (TypeScript Types)

Our smart contract utilizes a **Privacy-First Hybrid Architecture**. Instead of storing sensitive metadata like `preferences` on a public ledger, we store only a 32-byte cryptographic Verification Hash on-chain. The actual JSON data flows through a secure off-chain side-channel.

Here is the exact TypeScript type mapping you should use for your state management and UI scaffolding:

###  Types:
```typescript
/**
 * Room Environmental Preferences (The JSON string parsed)
 * Use this exact structure so backend IoT devices (Hue/Nest) can parse it.
 */
export interface RoomPreferences {
  temp: number;            // Celsius, e.g. 22.5
  brightness: number;      // 0-100 percentage
  light_color: string;     // Hex color code, e.g. "#FF5733"
  color_mode?: 'FOCUS' | 'RELAX' | 'SLEEP'; // Optional contextual modes
}

/**
 * Guest Identity Profile
 * Matches the Solana Anchor state exactly.
 */
export interface GuestProfile {
  owner: string;           // Pubkey string of the auth wallet
  name: string;            // Max 100 bytes
  loyaltyPoints: number;   // u64 casted to number
  stayCount: number;       // u32
  preferencesHash: number[]; // 32-byte array: SHA-256 hash of the off-chain RoomPreferences
}

/**
 * Firebase Synchronized State
 * The Node.js backend pushes updates here for instant UI reflect.
 */
export interface FirebaseGuestState {
  has_arrived: boolean;
  preferences: RoomPreferences;
  last_updated: number;   // timestamp
}
```

## 3. Firebase Listener Logic (Real-time UI Sync)

Since querying Solana RPCs for rapid UI feedback is often too slow and rate-limited, the UI *reads* from Firebase. Use this snippet to trigger your Next.js/React re-renders "instantly" when a user checks in or updates their ambient settings.

###  React Hook Example:
```typescript
import { useEffect, useState } from 'react';
import { getDatabase, ref, onValue, off } from 'firebase/database';
// Assuming you have an initialized Firebase App instance
// import { app } from '../lib/firebase'; 

export function useGuestRealtimeState(guestPda: string) {
  const [roomState, setRoomState] = useState<FirebaseGuestState | null>(null);

  useEffect(() => {
    if (!guestPda) return;

    const db = getDatabase(); // Assumes initialized firebase app
    // The backend uses '/rooms/guest_{pubkey}' pattern
    const guestRef = ref(db, `rooms/guest_${guestPda}`);

    // Subscribe to real-time changes
    const unsubscribe = onValue(guestRef, (snapshot) => {
      const data = snapshot.val();
      if (data) {
        console.log("Firebase sync triggered: UI Updating...");
        setRoomState(data as FirebaseGuestState);
      }
    }, (error) => {
      console.error("Firebase DB read error:", error);
    });

    // Clean up listener on unmount
    return () => off(guestRef, 'value', unsubscribe);
  }, [guestPda]);

  return roomState;
}
```

**Next Steps for Frontend:**
1. Implement the UI using the `RoomPreferences` interface.
2. **Hybrid State Mutation**: To update preferences in a privacy-preserving way:
   - **Step A:** Send the raw JSON string to our backend HTTP Gateway (`POST /api/preferences`).
   - **Step B:** Compute the precise SHA-256 hash of the exact JSON payload array on the client side.
   - **Step C:** Form your Solana transaction using `@coral-xyz/anchor`. Call `updatePreferences(Array.from(hash))` to lock the verified state on the blockchain.
3. Read the state updates via the `useGuestRealtimeState` hook to reflect UI changes instantly.

## 4. Production Integration Path (Current)

The integration now includes a production API gateway and persistent state:

### API ingress

- Endpoint: `POST /api/v1/voice-command`
- Backend file: `backend/src/api/server.ts`

Expected request body:

```json
{
  "guestPda": "<base58_pda>",
  "userInput": "Set room to comfort mode",
  "guestContext": {
    "name": "Guest Name",
    "loyaltyPoints": 1200,
    "history": ["prefers warm lighting"]
  }
}
```

### Hash-lock verification runtime

1. API stores pending command in Redis.
2. Solana account mutation updates `preferences_hash`.
3. `backend/src/listener.ts` reads on-chain hash.
4. `OrinAgent` rebuilds payload and generates local hash.
5. Hash compare:
   - Equal: publish MQTT (`MQTT_TOPIC`) + save `response.mp3`
   - Different: security alert (possible MITM/desync)

### Canonical hashing requirement

Hash generation and verification must use the same canonical function in:

- `backend/src/shared/hash.ts`

Do not implement ad-hoc JSON serialization in clients/services if hash-lock is expected to match.

### Runtime config

Use:

- `.env.example`
- `backend/.env.example`

Critical vars include:

- `RPC_ENDPOINT`
- `PROGRAM_ID`
- `GROQ_API_KEY`
- `DEEPGRAM_API_KEY`
- `MQTT_BROKER_URL`
- `MQTT_TOPIC`
- `REDIS_URL`
